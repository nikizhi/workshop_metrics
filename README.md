## **Введение в прикладные метрики**

Всем привет! На прошлом занятии мы начали знакомиться с **инструментами мониторинга** и подключили к нашему проекту **Prometheus**.

Давайте быстро вспомним общую схему работы с Prometheus, которая у нас получилась:
- FastAPI-приложение отдает метрики на `/metrics`
- Prometheus каждые 5 секунд забирает эти метрики
- Prometheus UI позволяет смотреть собранные данные

Также на предыдущем занятии мы создали файл `app/core/monitoring.py`, где настроили **автоматический** сбор HTTP-метрик с помощью новой библиотеки `prometheus-fastapi-instrumentator`.

```python
# app/core/monitoring.py

from fastapi import FastAPI
from prometheus_fastapi_instrumentator import Instrumentator

def setup_monitoring(app: FastAPI):
    Instrumentator().instrument(app).expose(
        app=app,
        endpoint="/metrics",
        include_in_schema=False
    )
```

Не менее важно то, что мы создали файл `prometheus.yaml` с конфигурацией Prometheus. То есть Prometheus сейчас ходит в наше FastAPI-приложение по адресу `api:8000/metrics` и забирает оттуда метрики.

```YAML
# prometheus.yaml

global:
  scrape_interval: 5s

scrape_configs:
  - job_name: "jobs_api"
    static_configs:
      - targets: ["api:8000"]
```

И чтобы Prometheus работал в едином окружении со всеми компонентами нашей системы, мы добавили сервис `prometheus` в `compose.yaml`. Благодаря этому мы можем обращаться к Prometheus UI через адрес http://127.0.0.1:9090.

```YAML
# compose.yaml

  prometheus:
    image: prom/prometheus
    restart: always
    depends_on:
      - api
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yaml:/etc/prometheus/prometheus.yml:ro
```

Также при работе с Prometheus UI на предыдущем занятии мы смотрели на метрику `http_requests_total`. Помним, что она формируется **автоматически** и показывает данные о всех HTTP-запросах к приложению. 

![Автоматическая метрика http_requests_total](img/http_requests_total.png)

Это очень полезная метрика, но есть важный момент, который мы сейчас обсудим.

Дело в том, что автоматические метрики показывают **техническую информацию** о запросах, это нам известно. Например, такие метрики могут показать, что был вызван эндпоинт `POST /jobs`. Но сам по себе вызов `POST /jobs` - это только **внешняя часть** работы приложения. Давайте вспомним, что происходит **внутри** `POST /jobs`:
1. Приложение принимает данные о новой задаче
2. Задача сохраняется в БД
3. Очищается summary-кэш
4. Задача отправляется на исполнение в Celery
5. Информация о созданной задаче возвращается клиенту

То есть во время работы одного запроса происходит сразу **несколько** важных событий. Но при этом автоматическая метрика от Prometheus покажет нам **только факт, что был вызван** `POST /jobs` **и все**. То есть мы не узнаем:
- Какого типа задача была создана?
- В какую очередь задача отправилась?

Именно для решения таких ситуаций и нужны **прикладные метрики**.

> 🔢 **Прикладные метрики** - числовые показатели, которые описывают события внутри бизнес-логики приложения.

На сегодняшнем занятии мы добавим **две** прикладные метрики:
1. `jobs_created_total` - сколько задач было создано через API
2. `jobs_sent_to_queue_total` - сколько задач было отправлено в очередь на исполнение

Связка этих метрик может быть весьма полезна. Например, представим ситуацию, что `jobs_created_total` растет, `jobs_sent_to_queue_total` растет **медленнее** или **не растет совсем**. Это может означать, что задачи создаются и сохраняются в БД, но по какой-то причине не доходят до очереди.

## **Создание прикладных метрик**

Теперь переходим к настройке прикладных метрик в нашем проекте и начнем с файла `app/core/monitoring.py`. Внутри есть функция `setup_monitoring()`, которая отвечает только за подключение **автоматических HTTP-метрик**.

Теперь добавим в файл собственные прикладные метрики. Для этого импортируем специальный класс `Counter` из библиотеки `prometheus_client`, которую мы установили на предыдущем занятии при установке `prometheus-fastapi-instrumentator`.

```python
# app/core/monitoring.py

from prometheus_client import Counter
```

Класс `Counter` позволяет создать одноименную метрику **внутри Prometheus**, давайте разберемся с ее определением.

> **Counter** - тип метрики в Prometheus, представляющий собой монотонно возрастающее значение, которое может **только увеличиваться** или **сбрасываться до нуля** при перезапуске приложения. Подходит для учета событий, количество которых может только увеличиваться.

В нашем проекте `Counter` удобно использовать в следующих случаях:
- Количество созданных задач
- Количество задач, отправленных в очередь

Итак, воспользуемся классом `Counter`, чтобы создать метрику `jobs_created_total`:

```python
# app/core/monitoring.py

jobs_created_total = Counter(
    name="jobs_created_total",
    documentation="Общее количество задач, созданных через API",
    labelnames=["job_type", "queue"]
)
```

- `jobs_created_total` - название объекта, через который мы будем обращаться к метрике
- `name="jobs_created_total"` - название метрики, которое будет видно в `/metrics` и Prometheus UI
- `documentation="Общее количество задач, созданных через API"` - описание метрики
- `labelnames=["job_type", "queue"]` - список `labels` внутри Prometheus

С понятием `labels` внутри Prometheus мы еще не знакомы, давайте разберемся что это.

В нашем проекте есть разные **типы задач**, которые описаны в классе `JobType` внутри `app/models/job.py`:

```python
# app/models/job.py

class JobType(StrEnum):
    HTTP_CHECK = "HTTP_CHECK"
    WORD_STATS = "WORD_STATS"
```

Также у нас есть разные **типы очередей**, которые описаны в классе `QueueName`, также внутри `app/models/job.py`:

```python
# app/models/job.py

class QueueName(StrEnum):
    DEFAULT = "default"
    PRIORITY = "priority"
```

Мы могли бы просто считать общее количество созданных задач в стиле: "Создано 10 задач". Но полезнее видеть более **подробную** картину:
- Создано задач с типом HTTP_CHECK в default-очереди: 4
- Создано задач с типом HTTP_CHECK в priority-очереди: 1
- Создано задач с типом WORD_STATS в default-очереди: 5
- Создано задач с типом WORD_STATS в priority-очереди: 8

Именно для этого в Prometheus используются `labels`.

> **Labels** - дополнительные признаки метрики, по которым Prometheus разделяет значения. Обычно выглядят как пары "ключ-значение".

Например, одна метрика может выглядеть вот так:
- jobs_created_total{job_type="HTTP_CHECK", queue="default"}

Здесь `jobs_created_total` - название метрики, а `job_type="HTTP_CHECK"` и `queue="default"` - labels в формате "ключ-значение".

Важно понимать, что labels - это **не отдельные метрики**. Это разные варианты одной и той же метрики.

С метрикой `jobs_created_total` разобрались, теперь создадим вторую метрику - `jobs_sent_to_queue_total`:

```python
# app/core/monitoring.py

jobs_sent_to_queue_total = Counter(
    name="jobs_sent_to_queue_total",
    documentation="Общее количество задач, отправленных в очередь",
    labelnames=["job_type", "queue"]
)
```

Обратите внимание - пока что мы **просто создали** метрики. Сами по себе они еще ничего не считают и никак не увеличиваются. Чтобы метрика начала считать события, нужно вызвать **специальный метод** (о нем будет чуть позже) в том месте программы, где происходит событие.

И еще немного обсудим labels, а точнее - их **ограничения**.

Хорошие labels - это значения, которые **повторяются** и имеют **ограниченное** количество вариантов.

Сейчас у нас хорошие labels (`job_type` и `queue`), потому что типов задач и очередей у нас мало.

Вот примеры плохих labels:
- `job_id`
- `request_id`
- `user_id`

Почему это плохие labels? Дело в том, что перечисленные значения почти всегда **уникальные**. Если добавить их в labels, то Prometheus начнет создавать **отдельный временной ряд** для каждой задачи, каждого запроса и каждого пользователя. Таких временных рядов может стать **слишком много** и это перегрузит систему мотиторинга. Поэтому запомним одно важное правило:

> В labels не стоит добавлять уникальные значения, например - ID.

## **Подключение метрик в POST /jobs**

Приступим к написанию кода для **накопления** метрик. Для этого обратимся к файлу `app/routes/job.py` и найдем функцию `create_job()`, именно она отвечает за создание задач через API.

Сначала добавим вверху импорт наших метрик из `app/core/monitoring.py`:

```python
# app/routes/job.py

from app.core.monitoring import jobs_created_total, jobs_sent_to_queue_total
```

Теперь нужно понять, где именно в коде `create_job()` происходят события, метрики которых мы хотим увеличивать. 

Найдем в коде функции строчку `job = await job_repo.create_job(session, payload)` - именно здесь задача создается и сохраняется в БД. Сразу после этой строки напишем код для увеличения значения метрики `jobs_created_total`:

```python
# app/routes/job.py

    jobs_created_total.labels(
        job_type=payload.job_type.value,
        queue=queue_name.value
    ).inc()
```

- `labels()` - метод, позволяющий указать значения labels для конкретного события
- `job_type=payload.job_type.value` - записываем тип созданной задачи
- `queue=queue_name.value` - записываем тип очереди, в которую будет отправлена задача
- `inc()` - увеличение счетчика метрики (по умолчанию - на 1)

Теперь найдем фрагмент, где задача отправляется в Celery - это строчка `run_job_task.apply_async(...)`. Сразу после нее напишем код для увеличения значения метрики `jobs_sent_to_queue_total`:

```python
# app/routes/job.py

    jobs_sent_to_queue_total.labels(
        job_type=payload.job_type.value,
        queue=queue_name.value
    ).inc()
```

Супер, мы внедрили накопление метрик в наш код! Обратите внимание, что в текущей функции `create_job()` хорошо видна **связь логов и метрик**. Коды логов и метрик находятся рядом, что логично. Ведь логи помогают отслеживать конкретные события, а метрики помогают увидеть общую картину по множеству событий. При всем этом у вас может появиться справедливый вопрос: "Не слишком ли много разного теперь в одной функции?". Да, теперь в `create_job()` содержатся:
- HTTP-слой
- Бизнес-логика
- Логирование
- Мониторинг

В рамках обучения это допустимо, т.к. мы видим цепочку:
> Произошло событие -> Записали лог -> Увеличили метрику

**В будущих уроках мы поговорим о том, как сделать эту архитектуру более чистой**.

## **Знакомство с PromQL**

Сейчас мы просто вводили название метрики и Prometheus показывал нам сырые значения.

Но Prometheus умеет не только выводить метрики напрямую. Он также позволяет выполнять **запросы к метрикам**. Для этого используется специальный язык запросов - **PromQL**.

> 🔍 **PromQL** - язык запросов для работы с метриками в Prometheus. Он позволяет выбирать и агрегировать данные временных рядов в реальном времени.

На текущем занятии мы посмотрим на **базовые запросы**, которые можно выполнять при помощи PromQL.

### Общее количество созданных задач

Начнем с запроса, благодаря которому мы получим **общее количество созданных задач**. Для этого воспользуемся специальным оператором `sum()`:

```promql
sum(jobs_created_total)
```

В результате выполнения увидим примерно следующее:

![Общее количество созданных задач](img/sum(jobs_created_total).png)

Этот запрос **складывает** все значения метрики `jobs_created_total`. То есть Prometheus покажет общее количество созданных задач **без разделения по типам и очередям**. Например, если мы создали 3 задачи, результат будет 3, если создали 4 - увидим 4.

### Количество созданных задач по типам

Теперь выполним запрос для получения **количества созданных задач с разделением по их типам**. Для этого вместе с оператором `sum()` воспользуемся оператором `by`, через который будет выполняться **группировка**:

```promql
sum(jobs_created_total) by (job_type)
```

В результате выполнения увидим примерно следующее:

![Количество созданных задач с разделением по их типам](img/sum(jobs_created_total)_by_(job_type).png)

Здесь мы говорим Prometheus: "*Сложи значения jobs_created_total и сделай группировку по job_type.*". 

В результате можно увидеть, сколько задач каждого типа было создано.

### Количество созданных задач по очередям

Теперь сделаем группировку задач по типу очереди:

```promql
sum(jobs_created_total) by (queue)
```

В результате выполнения увидим примерно следующее:

![Количество созданных задач по очередям](img/sum(jobs_created_total)_by_(queue).png)

Так мы видим, в какие очереди чаще всего уходили задачи.

### Количество отправленных задач по типам

Теперь выполним похожий запрос для второй метрики - получим количество отправленных задач по их типам:

```promql
sum(jobs_sent_to_queue_total) by (job_type)
```

В результате выполнения увидим примерно следующее:

![Количество отправленных задач по типам](img/sum(jobs_sent_to_queue_total)_by_(job_type).png)

Этот запрос покажет, сколько задач каждого **типа** было отправлено в очередь.

### Количество отправленных задач по очередям

Теперь сгруппируем отправленные задачи по типу очереди:

```promql
sum(jobs_sent_to_queue_total) by (queue)
```

В результате выполнения увидим примерно следующее:

![Количество отправленных задач по очередям](img/sum(jobs_sent_to_queue_total)_by_(queue).png)

Так можно увидеть, сколько задач ушло в **default**-очередь, а сколько — в **priority**.

### Группировка сразу по двум labels

Мы также можем выполнить запрос с группировкой сразу по **двум labels**:

```promql
sum(jobs_created_total) by (job_type, queue)
```

В результате выполнения увидим примерно следующее:

![Группировка сразу по двум labels](img/sum(jobs_created_total)_by_(job_type,queue).png)

Да, результат такой же, как и при `sum(jobs_created_total)`, но это потому что у нас **всего 2** разных labels. При большем количестве labels эта механика позволяет делать группировку **только по нужным** параметрам.
